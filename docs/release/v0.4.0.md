# iqdb-filter v0.4.0 — The Inverted Metadata Index

**From "test every row" to "look up the matches."** v0.3.0 made the strategy vocabulary actionable; v0.4.0 adds the data structure behind `PreFilter`. An opt-in, per-field `MetadataIndex` turns a selective `Eq` / `In` predicate into a candidate key set — a superset of the true matches — so a consumer scores a handful of rows instead of all of them, and the strategy selector gets a count-based selectivity estimate sharp enough to trust.

## What is iqdb-filter?

The metadata-filtering layer of the iQDB vector database. It owns one definition of what a `Filter` means and, as of this release, one way to *accelerate* it: the inverted index every metadata-aware index can share rather than re-deriving.

## What's new in 0.4.0

### `MetadataIndex` — opt-in, per-field, superset-correct

`build` indexes only the fields you name — per-field opt-in, because indexing a field nobody filters on is pure write-amplification. `candidates` resolves a validated filter to the keys that *might* match; the consumer confirms exactness with `evaluate`.

```rust
use iqdb_filter::{FilterEvaluator, MetadataIndex};
use iqdb_types::{Filter, Metadata, Value};

let rows = [
    (0_usize, [("lang".to_string(), Value::String("rust".into()))].into_iter().collect::<Metadata>()),
    (1, [("lang".to_string(), Value::String("go".into()))].into_iter().collect::<Metadata>()),
    (2, [("lang".to_string(), Value::String("rust".into()))].into_iter().collect::<Metadata>()),
];
let index = MetadataIndex::build(&["lang"], rows.iter().map(|(k, m)| (*k, Some(m))));

let evaluator = FilterEvaluator::new(Filter::eq("lang", Value::String("rust".into())))
    .expect("valid filter");
let mut hits = index.candidates(&evaluator).expect("indexed field resolves");
hits.sort_unstable();
assert_eq!(hits, [0, 2]);
```

### The correctness contract

When `candidates` returns `Some(set)`, **every record the evaluator accepts is in `set`** — false positives are allowed (the consumer filters them), false negatives never happen. This is pinned by a property test that checks, across thousands of random filters and records, that no true match is ever missing. It returns `None` for the predicates an inverted index can't bound — ranges, `Neq`, `Not`, `Float` literals (IEEE equality would risk dropping a match), and non-indexed fields — which simply means "scan everything." `And` intersects whatever children resolve and leaves the rest to `evaluate`; `Or` resolves only if every branch does.

### Index-backed selectivity

`MetadataIndex::estimate_selectivity` upgrades the structural estimate with real posting counts: an indexed `Eq` / `In` leaf contributes its true `matches / total` fraction, with the structural fallback elsewhere. `StrategySelector::choose_with_index` feeds that sharper number into the pre/post decision.

```rust
use iqdb_filter::{FilterEvaluator, MetadataIndex};
use iqdb_types::{Filter, Metadata, Value};

let rows = [
    (0_usize, [("status".to_string(), Value::Int(1))].into_iter().collect::<Metadata>()),
    (1, [("status".to_string(), Value::Int(2))].into_iter().collect::<Metadata>()),
    (2, [("status".to_string(), Value::Int(2))].into_iter().collect::<Metadata>()),
    (3, [("status".to_string(), Value::Int(2))].into_iter().collect::<Metadata>()),
];
let index = MetadataIndex::build(&["status"], rows.iter().map(|(k, m)| (*k, Some(m))));

let eq = FilterEvaluator::new(Filter::eq("status", Value::Int(1))).expect("valid");
assert!((index.estimate_selectivity(&eq) - 0.25).abs() < 1e-9); // the real 1-in-4
```

## Performance

On the reference machine, over a 1,000-row index on two fields:

| Path | Time |
|------|------|
| `build` (1,000 rows, 2 fields) | ~56 µs |
| `candidates` (`Eq`, ~250 matches) | ~2.4 µs |
| `candidates` (`And` of two equalities) | ~7.8 µs |

Numbers are from the `criterion` bench (`cargo bench`).

## What's deferred (and why)

Only **`InFilter` pushdown** into graph traversal remains — pruning HNSW/IVF branches that can't yield a surviving candidate. It needs a real approximate-index consumer to validate recall before it can be built honestly, and being purely additive it can also ship in a post-1.0 MINOR release. Histogram/sketch estimators over the index are a possible future refinement, not a requirement: exact posting counts already give an accurate equality/membership estimate. See `dev/ROADMAP.md`.

## Breaking changes

**Pre-1.0 additive release.** All new surface; nothing from v0.3.0 changed shape.

## Verification

Covered by unit tests (resolution rules per predicate, empty/`None` distinctions, real-count selectivity), property tests (the superset contract; index selectivity is always a probability), and benchmarks. The same gates run across the CI matrix (Linux, macOS, Windows) on stable and the 1.87 MSRV:
