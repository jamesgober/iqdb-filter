# iqdb-filter v0.5.0 — Hardening & API Freeze

**Stop adding, start committing.** v0.2–v0.4 built the surface — evaluator, strategy selection, inverted index. v0.5.0 adds no new public API. Instead it proves the surface is sufficient and robust, then freezes it: from here, only additive changes until 2.0.

## What is iqdb-filter?

The metadata-filtering layer of the iQDB vector database — one definition of what a `Filter` means, the scan helpers and inverted index that accelerate it, and the selectivity-driven selector that chooses how to apply it. As of this release, a committed, stable surface.

## What's new in 0.5.0

### Consumer-simulation suite

A filtered top-`k` searcher built **only** on the public API — `MetadataIndex` to narrow candidates, `evaluate` to confirm, `prefilter` / `postfilter` to scan, the selector to choose a strategy. It runs every filter shape through two paths, index-accelerated and full-scan, and asserts they return **exactly the same result**. That is the operational proof of the index's superset contract: if the index ever dropped or invented a match, these tests fail. It is also the proof that the public surface is enough to build a real consumer — no private hooks required.

### Fuzz targets

Two `cargo-fuzz` targets under `fuzz/`, wired into CI:

- **`robustness`** — `FilterEvaluator::new` always returns a typed `Result` (never panics, never overflows) on arbitrary filters, and `evaluate` / `prefilter` never panic on arbitrary metadata, including `NaN` / `Inf` floats and absent fields.
- **`superset`** — the `MetadataIndex::candidates` contract (every true match is a candidate) holds on unbounded, adversarial input.

Both inputs are generated from `arbitrary`-derived mirrors of the `iqdb-types` filter and value types, with conversion depth capped so building the tree can't itself overflow. In a local smoke run both targets cleared **>250k executions each with zero crashes**.

### API freeze

The public surface is now committed: **only additive (MINOR) changes until 2.0.** The frozen surface is recorded in `dev/ROADMAP.md`. The design kept the right seams open for the one deferred feature — `FilterStrategy` is `#[non_exhaustive]` and the selectivity / candidate paths are standalone functions — so `InFilter` pushdown, when it lands, is a non-breaking addition.

## Breaking changes

**None.** This release adds tests, fuzz targets, and a stability commitment; the public API is unchanged from v0.4.0.

## What's deferred

Only **`InFilter` pushdown** into graph traversal, which needs a real approximate-index consumer (`iqdb-hnsw` / `iqdb-ivf`) to validate recall. Being additive, it can ship pre- or post-1.0 without a breaking change. See `dev/ROADMAP.md`.

## Verification

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo deny check

# Fuzz (nightly + cargo-fuzz):
cargo +nightly fuzz build
cargo +nightly fuzz run robustness -- -max_total_time=30
cargo +nightly fuzz run superset   -- -max_total_time=30
```

Verified across the CI matrix (Linux, macOS, Windows) on stable and the 1.87 MSRV. MSRV: Rust 1.87.

## What's next

- **RC (0.6.x → 0.9.x)** — soak against the first real consumer (`iqdb-flat`), final benchmarks, doc polish; then 1.0. The API is frozen and hardened, so this is soak time, not new surface.
